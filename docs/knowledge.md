## 技術知見
### このアプリがサーバー不要だった理由
- `/api/check` の正誤判定は「答えのパターン」を使わず、ユーザー正規表現が examples 全マッチ・non_examples 全非マッチかを見るだけ。よって判定はクライアントで完結できる。
- 答えのパターンはライフ切れ時の表示用途のみ。`solutions.html` で元々全公開しているので隠す価値も低い。

### Python re ↔ JS RegExp 移植
- `re.fullmatch(pat, s)` ≡ `new RegExp('^(?:'+pat+')$').test(s)`。
- 二重アンカー（ユーザーが ^ $ を含む）でも等価。
- 今回使う正規表現機能はすべて JS 互換。

## ハマりポイント
- テストは独自 `it()` ハーネスで書かれており `node --test` では MODULE_NOT_FOUND になる。`node tests/regex-core.test.js` で直接実行する。
- 判定/出題のコアは `js/regex-core.js` に純関数（`judgePattern` / `selectProblem` / `normalizeLevel`）として切り出し済み。Node からも window からも使えるユニバーサル定義（IIFE + module.exports 両対応）。
- 実装は Codex に委譲、Claude がクロスレビュー＋テスト実行＋コミットを担当（A モード）。Codex 生成の空 `.agents/` `.codex/` は .gitignore で除外。

## 学習済み概念
- 2026-05-31: **静的化できる条件** — リクエストごとのサーバー処理（計算・DB・秘匿保持）が不要なら、静的ホスティング（用意済みファイルを返すだけ）で足りる。本アプリは正誤判定が「答えのパターン」不要＝例文マッチ可否だけで成立するため該当。（理解度テスト3問正解）
- 2026-05-31: **静的 vs サーバー型の本質** — 静的は「用意済みファイルを返す」だけ。JS はブラウザ上で動くので静的でもクライアント処理は可能。動かないのは「サーバー側コード」。
- 2026-05-31: **静的化のトレードオフ** — 「答えをサーバーで隠す」性質を失う。問題データが JS に乗り、開発者ツールで答えが見える。学習用ゲーム＆解答一覧で元々公開のため実害小。
